Migration Scope: What gets ported to Three.js?

We're moving the sphere rendering to Three.js. But the current Canvas 2D renderer also draws markers, paths, arcs, regions, and graticule. How much do we migrate?

A

Hybrid: Three.js sphere + HTML/CSS overlay for everything else

Three.js renders only the earth (textures, atmosphere, lighting). Markers, callouts, paths, arcs, and regions are drawn as HTML/CSS elements positioned via projectPointToClient(). The existing projection math stays, just outputs to DOM instead of canvas.

Pros

  • Callouts are naturally HTML — easy styling, accessibility, i18n
  • Smallest migration — only sphere body changes
  • CSS animations for callout transitions

Cons

  • Arcs/paths/regions as HTML are tricky (SVG overlay needed)
  • Two coordinate systems to sync
B

Hybrid: Three.js sphere + Canvas 2D overlay for lines, HTML for callouts

Three layers: (1) Three.js WebGL canvas for the earth, (2) Canvas 2D overlay for paths/arcs/regions/graticule (existing drawing code), (3) HTML overlay for marker dots and callouts.

Pros

  • Existing path/arc/region drawing code reused as-is
  • Callouts as HTML — rich content, accessible
  • Clean separation of concerns

Cons

  • Three render layers to coordinate
  • Slight complexity in z-ordering
C

Full Three.js: everything in WebGL

Port all rendering to Three.js: earth sphere, markers as 3D sprites/billboards, arcs/paths as Line geometry, regions as extruded meshes. Callouts use CSS2DRenderer (Three.js built-in HTML overlay).

Pros

  • Single rendering pipeline — no sync issues
  • Better depth sorting (markers behind globe auto-hidden)
  • Future-proof for 3D marker models
  • Aligns with PRD's full Three.js vision

Cons

  • Largest migration — most existing drawing code replaced
  • Callouts still use CSS2DRenderer (HTML), so we get HTML callouts anyway

My recommendation: Option C (Full Three.js) — since we're already adding the dependency, go all-in. Three.js's CSS2DRenderer gives us HTML callouts for free, depth sorting is automatic, and we avoid the complexity of syncing multiple render layers. This matches the PRD's original intent.
